# Auth 用户认证服务 — 风险分析报告

> **项目**：`auth-web`（软件测试课程被测试应用）
> **文档版本**：v1.1
> **编写日期**：2026-05-22

## 一、项目简述

**Auth 用户认证服务** 是一个独立的用户身份认证子系统，作为软件测试课程作业中 AutoTestDesign 工具的被测目标应用（SUT）。系统围绕「注册—登录—会话管理—资料维护—密码找回」构建完整的用户身份生命周期。

### 1. 核心目标

- 提供 **用户注册、登录、密码重置、个人资料更新** 等标准认证能力
- 通过 **邮箱验证码** 保障注册与找回密码流程的可信性
- 采用 **JWT + Redis 双校验** 实现会话管理与单点登录（新登录踢旧 Token）
- 提供 **接口清晰、边界明确** 的 REST API，便于测试设计与自动化验证

### 2. 系统用户视角支持

本系统为**单一角色**（普通注册用户）：

| 用户操作   | 说明                                         |
| ---------- | -------------------------------------------- |
| 注册       | 填写用户名、邮箱、密码，邮箱验证码校验后入库 |
| 登录       | 支持用户名或邮箱 + 密码，返回 JWT Token      |
| 忘记密码   | 邮箱验证码校验后重置密码                     |
| 个人资料   | 登录后更新姓名、电话、主页链接               |
| 唯一性校验 | 注册前可检查用户名/邮箱是否已占用            |

### 3. 技术架构与实现细节

| 层级   | 技术选型                             | 功能说明                                     |
| ------ | ------------------------------------ | -------------------------------------------- |
| 前端   | Vue 3 + Vite + TypeScript + Naive UI | 登录、注册、重置密码、个人资料页面           |
| 后端   | Spring Boot 3.5 + Spring Security    | RESTful API、JWT 鉴权、全局异常处理          |
| 数据层 | PostgreSQL + MyBatis-Plus            | 用户表持久化、唯一索引约束                   |
| 缓存层 | Redis + Lua 脚本                     | 验证码存储与原子校验删除、JWT Token 会话缓存 |
| 邮件   | Spring Mail（SMTP）                  | 注册/重置密码验证码投递                      |
| 鉴权   | JWT（jjwt 0.12）+ Redis 二次校验     | Bearer Token + 服务端会话一致性              |
| 密码   | BCrypt（Spring Security）            | 加盐哈希存储与校验                           |

**被测对象**：`{API_BASE_URL}` 下的 REST 接口（前缀 `/api/zdmj`）。`{API_BASE_URL}` 为项目配置的 API 基址（见 `README.md` 或 Apifox 环境变量），**不在本文档中写死主机地址**。测试以 **接口功能与安全** 为主，通过 Apifox、REST Assured、curl 等工具直接调用。

### 4. 数据与功能模块概览

| 模块            | 主要接口                                  | 认证要求 |
| --------------- | ----------------------------------------- | -------- |
| 用户注册        | `POST /users`                           | 否       |
| 用户登录        | `POST /users/login`                     | 否       |
| 发送验证码      | `POST /users/verification-codes?email=` | 否       |
| 重置密码        | `PUT /users/password`                   | 否       |
| 用户名/邮箱校验 | `GET /users/validation/*`               | 否       |
| 查询用户        | `GET /users/{id}`                       | 是       |
| 更新当前用户    | `PUT /users/me`                         | 是       |

数据库核心为 **`users` 单表**，字段含 `username`、`password`（BCrypt）、`email`、`name`、`phone`、`website` 及时间戳。

### 5. 测试重要性说明

- **账户安全**：密码存储、Token 签发与校验、越权访问
- **业务可用性**：注册/登录成功率、验证码可达性
- **测试重点**：登录接口 `POST /users/login` 为核心被测范围

风险分析围绕架构依赖、认证链路、验证码机制与会话管理展开，为测试计划提供优先级依据。

## 二、系统架构相关技术风险

| 风险名称                | 描述                                 | 影响范围           | 风险等级 | 预防策略                                      |
| ----------------------- | ------------------------------------ | ------------------ | -------- | --------------------------------------------- |
| MyBatis-Plus 映射异常   | 实体与表结构不一致导致查询失败       | 数据访问、业务逻辑 | 中       | 对照 `schema.sql`；接口集成测试验证返回字段 |
| Redis 依赖异常          | Redis 不可用时验证码、JWT 会话失败   | 全链路认证         | 高       | 接口测试观察错误码与 `/actuator/health`     |
| JWT 密钥配置不当        | 密钥泄露或过短导致 Token 被伪造      | 全站鉴权           | 高       | 通过 Token 篡改用例验证签名校验               |
| JWT 与 Redis TTL 不一致 | Token 过期时间与 Redis 缓存 TTL 脱节 | 会话异常           | 中       | 测试 Token 过期与踢下线场景                   |
| CSRF 防护关闭           | 纯 API + Bearer Token 场景可接受     | 跨站伪造           | 低       | 确认 Token 不存入 Cookie                      |
| CORS 配置过宽           | `allowedOriginPatterns` 含 `*`   | 跨域滥用           | 中       | 跨域请求抽查                                  |

## 三、功能模块详细风险分析

### 3.1 用户注册与登录模块

| 风险项               | 描述                                               | 影响             | 风险等级 |
| -------------------- | -------------------------------------------------- | ---------------- | -------- |
| 邮箱验证码不稳定     | SMTP 发送失败、延迟、进垃圾箱                      | 注册/重置失败    | 中       |
| 验证码暴力破解       | 6 位数字无频控可被枚举                             | 绕过邮箱验证     | 高       |
| 验证码发送无频率限制 | 同邮箱可频繁请求                                   | 邮件轰炸         | 高       |
| 用户名/邮箱枚举      | 公开校验接口可探测已注册账号                       | 隐私泄露         | 中       |
| 登录凭证统一错误提示 | 用户不存在与密码错误均返回 `USER_PASSWORD_WRONG` | 需验证提示一致性 | 低       |
| 用户名含 `@` 误判  | 登录用 `contains("@")` 区分邮箱与用户名          | 登录失败         | 中       |
| 密码明文传输         | 未使用 HTTPS 时密码可被窃听                        | 中间人攻击       | 高       |
| 密码明文存储         | 若未 BCrypt 则数据库泄露致命                       | 账户失守         | 高       |
| 并发注册竞态         | 同时注册相同用户名                                 | 重复注册或 500   | 中       |
| 旧 Token 踢下线      | 新登录后旧 Token 应失效                            | 会话策略         | 中       |

**应对措施：**

- 测试验证 BCrypt 存储、验证码一次性使用、登录/注册边界参数
- 建议开发侧补充：验证码发送频率限制、错误次数锁定

### 3.2 验证码与邮件模块

| 风险项             | 描述                      | 风险等级 | 应对措施                          |
| ------------------ | ------------------------- | -------- | --------------------------------- |
| 验证码可预测       | 使用 `java.util.Random` | 中       | 统计与边界测试                    |
| 验证码明文邮件     | 邮件正文含明文验证码      | 低       | 流程测试                          |
| Redis 键无发送间隔 | 可频繁触发邮件            | 高       | 接口频率测试                      |
| Lua 脚本执行失败   | 校验逻辑异常              | 中       | 正确/错误/过期/重复使用用例       |
| SMTP 发送失败      | 邮件服务异常              | 中       | 验证 `CAPTCHA_SEND_FAILED` 响应 |

### 3.3 JWT 会话与鉴权模块

| 风险项                         | 描述                                 | 风险等级 | 应对措施                  |
| ------------------------------ | ------------------------------------ | -------- | ------------------------- |
| Token 泄露（XSS/localStorage） | 前端 Token 存 localStorage           | 高       | XSS 与安全走查            |
| 越权访问他人资料               | `GET /users/{id}` 归属校验         | 高       | A 用户 Token 访问 B 的 id |
| 无 Token 访问受保护接口        | 应返回 401                           | 中       | 鉴权用例全覆盖            |
| Token 格式兼容                 | 支持 Bearer 与无前缀                 | 低       | 两种格式回归              |
| 踢下线策略                     | Redis 中 Token 被替换后旧 Token 失效 | 中       | 双次登录对比测试          |
| `GET /users/{id}` 无归属校验 | 任意登录用户可查任意 id              | 中       | 明确规则并测试            |

### 3.4 密码重置与个人资料模块

| 风险项                | 描述                    | 风险等级 | 应对措施                          |
| --------------------- | ----------------------- | -------- | --------------------------------- |
| 重置密码仅凭验证码    | 验证码泄露可接管账户    | 高       | 验证码安全与重置流程测试          |
| 重置后旧 Token 仍有效 | 未清除 Redis JWT        | 高       | 重置后旧 Token 访问 `/users/me` |
| 资料字段缺少校验      | name/phone/website 边界 | 中       | 超长、特殊字符、XSS 载荷          |
| 部分更新逻辑          | null 与空字符串行为     | 低       | 边界用例                          |

### 3.5 测试执行相关风险

| 风险项           | 描述                            | 风险等级 | 解决策略                  |
| ---------------- | ------------------------------- | -------- | ------------------------- |
| 测试数据污染     | 共享库导致账号互相干扰          | 中       | 使用带独立前缀的测试账号  |
| 接口基址不一致   | Apifox 与前端/脚本 baseURL 不同 | 中       | 统一为 `{API_BASE_URL}` |
| 前端默认测试账号 | 页面预填凭据（已清理）          | 低       | 测试时使用自建账号        |

### 3.6 第三方依赖风险

| 组件       | 风险                 | 建议                                       |
| ---------- | -------------------- | ------------------------------------------ |
| PostgreSQL | 数据不一致或约束冲突 | 通过注册/查询接口验证持久化结果            |
| Redis      | Token/验证码丢失     | 登录与会话相关接口回归                     |
| SMTP       | 验证码不可达         | `POST /verification-codes` 成功/失败路径 |
| jjwt       | 库升级导致解析失败   | Token 全流程回归                           |

### 3.7 未来扩展风险

| 风险项           | 描述                     | 风险等级 | 对策建议         |
| ---------------- | ------------------------ | -------- | ---------------- |
| 用户量剧增       | 登录接口成为热点         | 中       | 性能测试（可选） |
| 多因素认证缺失   | 仅密码+邮箱验证码        | 中       | 预留 MFA         |
| OAuth/第三方登录 | 未支持，扩展引入新攻击面 | 低       | 模块化认证       |

## 五、风险控制流程建议

### 测试前

- 基于风险矩阵确定优先级：**登录、注册、JWT 鉴权、验证码、密码重置**
- 统一 API 基址：`{API_BASE_URL}`
- 准备独立测试账号（建议使用可识别的命名前缀）

### 测试执行中

- **接口测试（核心）**：8 个 API 的正常路径、边界、错误码、鉴权头
- **工具**：Apifox
- **前端**：四页面走查，验证与接口契约一致
- **安全**：越权、Token 篡改、验证码重放等

### 测试后

- 汇总接口覆盖率与缺陷
- 不在报告中写入真实密码

## 六、结论

本报告对 Auth 用户认证服务的接口层风险进行了系统识别，**B1/B2/B3、D1/D2、E1/E2、A1/A2** 为最高优先级，应通过针对服务器 API 的功能与安全测试重点覆盖。测试重心为接口契约与业务逻辑正确性。
