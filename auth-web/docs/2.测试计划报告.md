## 1 项目范围

### 1.1 被测应用背景

**Auth 用户认证服务**（`auth-web`）是软件测试课程期末项目的**被测目标应用**，提供用户身份全生命周期能力：**注册 → 登录 → 会话保持 → 资料维护 → 密码找回**。

系统采用前后端分离架构：

- **后端**：Spring Boot 3.5、Spring Security、MyBatis-Plus、PostgreSQL、Redis、JWT
- **前端**：Vue 3、Vite、TypeScript、Naive UI、Axios

**被测 API 基址**：`{API_BASE_URL}`（前缀 `/api/zdmj`）。具体主机地址由项目配置或测试环境提供，本文档使用占位符，不写入明文 IP/域名。

**AutoTestDesign 在本项目测试活动中的角色**：

1. 导入 `auth-web` 需求（CSV / 文本 / 手工输入）
2. 结构化解析输入字段、边界、条件与预期动作
3. 结合《风险分析报告》进行风险评分与测试优先级排序
4. 按 ISO 29119-4 黑盒技术（等价类划分 EP、边界值分析 BVA、决策表 DT）生成测试用例
5. 支持测试人员**交互式审查、修订**覆盖项、策略与用例（作业"主要内容"要求）
6. 导出 JSON / Excel 测试工件，供 Apifox 执行；白盒用例由 JUnit 实现

### 1.2 测试活动总体目标

| 目标编号 | 目标描述                       | 验收标准                                                          |

### 3.2 各套件详细设计

#### 3.2.1 TS-01 用户注册与登录测试套件

**测试目标：**

确保用户注册与登录功能在不同输入场景下行为稳定、安全合规，验证凭证校验、验证码绑定、密码加密存储及新登录踢旧 Token 等核心逻辑，防止账号滥用与身份伪造。

**测试覆盖范围与测试层级：**

| 测试层级      | 主要测试对象                           | 关键测试点                                                | 涉及风险点                          | 测试方法与工具    |
| ------------- | -------------------------------------- | --------------------------------------------------------- | ----------------------------------- | ----------------- |
| 组件 / 白盒级 | 注册 / 登录业务逻辑、密码加密校验      | 用户名 / 邮箱格式校验；密码长度边界；用户名与邮箱识别逻辑 | B7 用户名含 `@` 误判、B4 密码存储 | JUnit 5 + Mock    |
| 组件 / 接口级 | `POST /users`、`POST /users/login` | 有效 / 无效等价类；空字段、非法邮箱；正确登录返回 token   | B1 验证码暴力破解、B3 密码明文传输  | Apifox + EP / BVA |
| 集成级        | 注册 → 登录 → Token 签发             | 新用户注册后可登录；新登录后旧 Token 失效                 | D5 踢旧 Token、B5 验证码不稳定      | Apifox 场景 + DT  |
| 系统级        | 注册页 + 登录页联调                    | 页面提交与 API 契约一致；错误提示清晰                     | NF-03 异常提示                      | Chrome 手工走查   |

**测试优先级：** Critical（关键）

**覆盖率目标建议：**

- 注册 / 登录接口路径覆盖率 ≥ 100%
- EP / BVA / DT 用例合计 ≥ 15 条，错误码覆盖 ≥ 80%
- 注册 / 登录业务逻辑核心分支白盒覆盖率 ≥ 80%

**补充说明：**

- 安全测试重点验证 HTTPS 下密码传输、密码加密存储（B4）
- 登录凭证组合（用户名 / 邮箱 × 密码正确 / 错误）须用决策表覆盖
- 用户名含 `@` 的特殊场景（B7）须单独验证

#### 3.2.2 TS-02 验证码与邮件测试套件

**测试目标：**

验证验证码生成、发送、校验及邮件投递流程的正确性与健壮性，确保验证码一次性使用、格式合法，并在 SMTP / Redis 异常时返回明确错误码。

**测试覆盖范围与测试层级：**

| 测试层级      | 主要测试对象                       | 关键测试点                            | 涉及风险点                      | 测试方法与工具    |
| ------------- | ---------------------------------- | ------------------------------------- | ------------------------------- | ----------------- |
| 组件 / 白盒级 | 验证码生成与校验逻辑               | 6 位数字格式；校验后原子删除          | B2 发送无频率限制               | JUnit 5 + Mock    |
| 组件 / 接口级 | `POST /users/verification-codes` | 合法 / 非法邮箱；未注册邮箱；重复发送 | B2 发送无频率限制、C3 SMTP 失败 | Apifox + EP / BVA |
| 集成级        | 发码 → 注册 / 重置消费            | 验证码一次性；过期码拒绝              | B1 验证码暴力破解               | Apifox 场景       |
| 验收级        | 邮箱实际收信                       | 验证码可达、正文格式正确              | B5 验证码不稳定                 | 邮箱客户端 + 手工 |

**测试优先级：** High（高）

**覆盖率目标建议：**

- 验证码接口 EP / BVA 用例 ≥ 8 条
- 错误路径（非法邮箱、SMTP 失败）覆盖 ≥ 90%

**补充说明：**

- 验证同邮箱频繁请求的频率限制（B2）
- SMTP 不可达时须返回明确的验证码发送失败错误码

#### 3.2.3 TS-03 JWT 鉴权与会话测试套件

**测试目标：**

确保 JWT 签发、校验、Redis 二次校验及受保护接口访问控制正确，防止 Token 泄露、越权访问与无 Token 绕过，验证踢下线与会话一致性。

**测试覆盖范围与测试层级：**

| 测试层级      | 主要测试对象                           | 关键测试点                                      | 涉及风险点                       | 测试方法与工具   |
| ------------- | -------------------------------------- | ----------------------------------------------- | -------------------------------- | ---------------- |
| 组件 / 白盒级 | JWT 鉴权过滤器、JWT 签名校验模块       | Token 解析；篡改签名拒绝；Bearer 格式兼容       | A2 JWT 密钥 / 签名               | JUnit 5          |
| 组件 / 接口级 | `GET /users/{id}`、`PUT /users/me` | 无 Token / 错误 Token / 有效 Token / 篡改 Token | D1 Token 泄露、D3 无 Token 401   | Apifox + EP      |
| 集成级        | 用户 A Token 访问用户 B 资源           | 越权访问拦截；Redis 会话与 Token 一致           | D2 越权访问、D5 踢下线           | Apifox + DT      |
| 系统级        | 登录 → 鉴权 → 登出 / 重新登录        | Token 失效后受保护接口 401                      | E1 重置后旧 Token、A1 Redis 异常 | Apifox 场景 + ST |

**测试优先级：** Critical（关键）

**覆盖率目标建议：**

- 受保护接口鉴权用例覆盖率 = 100%
- Token 异常场景（无 / 错 / 篡改 / 过期）≥ 4 条
- 越权访问组合（用户 A × 资源 B）≥ 2 条

**补充说明：**

- 越权（D2）、踢下线（D5）为本套件安全测试重点
- 状态转换：未登录 → 已登录 → Token 失效 → 重新登录

**补充说明：**

- 本轮不引入 Selenium / Playwright，以 Chrome 手工走查为主

### 3.3 AutoTestDesign 设计工作流

测试套件与用例的设计遵循工具要求的交互式流程：

```
概念理解 → 覆盖项识别 → 覆盖策略与方法选定 → 测试用例生成
    → 提示词设计 → 结果分析（用例 ↔ 覆盖项 ↔ 策略映射）
    → 人工审查修订 → 导出 JSON/Excel → 导入执行框架
```

**人工审查要点**（工具生成后必做）：

- 补充风险报告中的 High / Critical 项（如 B2 频率限制、D2 越权）
- 删除重复或不可执行用例
- 修正预期结果与系统错误码不一致项
- 保留修订记录，满足作业对"交互式验证有效性"的要求

## 6 选定测试框架及其理由

| 层次         | 框架 / 工具                | 理由                                                 |
| ------------ | -------------------------- | ---------------------------------------------------- |
| 测试设计     | AutoTestDesign             | 需求结构化、用例生成与导出                           |
| 黑盒（API）  | Apifox                     | 快速执行 8 个 API 用例，可导入工具导出结果           |
| 白盒（单元） | JUnit 5 + Spring Boot Test | 与 Spring Boot 同栈，覆盖 Service / 工具类逻辑与分支 |
| UI 验证      | Chrome 手工走查            | 4 页主流程，周期内成本可控                           |
| 辅助         | 邮箱客户端                 | 注册 / 重置流程收取验证码                            |

**选型原则**：黑盒以 Apifox 为主；白盒针对注册登录、JWT 鉴权等核心业务模块编写 JUnit 单元测试；UI 范围小，不引入 Selenium / Playwright。AutoTestDesign 导出用例 → Apifox 执行 → JUnit 补充白盒覆盖 → 结果回填，完成追溯。
